# Chatroom 聊天室 — 需求文档

> v2.1 | 2026-05-06

## 1. 项目概述

构建一个类QQ的实时聊天室应用，支持用户注册登录、好友管理、一对一私聊、群组聊天。系统具备高并发处理能力，确保消息即时性、一致性和可靠性。支持多个AI机器人以不同语言风格和情绪模式同时在线参与聊天。

## 2. 功能需求

### 2.1 用户模块
- 用户名+密码注册，BCrypt加密存储
- JWT Token认证，有效期7天
- 查看/修改昵称、头像
- 实时在线/离线状态

### 2.2 好友模块
- 通过用户名/昵称搜索用户，发送好友申请
- 接受/拒绝申请，解除好友关系
- 好友列表展示及在线状态

### 2.3 一对一聊天
- WebSocket实时推送，消息即时送达
- 分页拉取历史消息，保留30天
- 消息引用回复、2分钟内可撤回
- 送达/已读状态追踪

### 2.4 群组聊天
- 创建群组、邀请/移除成员（群主/管理员权限）、退出群组
- 群内广播消息，所有在线成员实时接收

### 2.5 AI机器人
- 多个机器人以独立WebSocket账号接入，同时在线
- 每个机器人拥有不同语言风格和情绪模式（Skill）
- 支持关键词触发、@提及、全自动回复三种模式
- 预期同时在线机器人 ≥ 20个，错误率 < 0.1%
- **导入聊天记录生成机器人**: 上传聊天记录文件（JSONL/JSON/TXT），系统自动分析每个发送者的语言风格和情绪模式，一键生成对应机器人
- **Skill生命周期**: 创建 → 预览 → 绑定机器人 → 上线/下线 → 版本更新/回滚
- **Skill入口**: UI与API两种方式同时提供
- **AI导入**: 支持接入多厂商API Key（不同模型/不同Key），每个Bot独立配置
- **QQ/微信导入**: 优先支持JSON/TXT导出格式，导入后可映射到Skill与Bot

## 3. 非功能需求

### 3.1 性能
- 注册用户 10万+，DAU 5000+，峰值WebSocket并发 1000+
- 日均消息 10万+，峰值QPS 500+
- API响应 P95 < 200ms，消息端到端延迟 P95 < 500ms
- 水平扩展：无状态服务 + K8s多副本 + 负载均衡

### 3.2 消息可靠性
- **At-Least-Once**: 推送 + ACK确认 + 超时重推
- **Exactly-Once**: clientMessageId (UUID) 去重 + DB唯一索引
- **全局有序**: 数据库自增ID保证顺序
- **离线补全**: 上线后从DB拉取未读消息
- 消息状态：SENDING → SENT → DELIVERED → READ / FAILED

### 3.3 安全
- 密码BCrypt (cost=10)，JWT HMAC-SHA256
- API/WebSocket权限控制 + 输入校验
- XSS/SQL注入防护

### 3.4 扩展性
| 维度 | 方案 |
|------|------|
| 应用层 | 无状态服务 + K8s + Nginx ip_hash |
| WebSocket | Redis Pub/Sub跨节点路由 |
| 数据库 | 读写分离 → 分库分表 (user_id取模) |
| 缓存 | Redis Cluster + Sentinel高可用 |
| 消息队列 | RabbitMQ集群 |

扩容触发: 单节点CPU > 70% (5min) / WS连接 > 800/节点

### 3.5 数据保留
- 消息保留30天，每日凌晨3:00定时清理
- 清理前自动提取对话对供Skill蒸馏

### 3.6 测试与可验证性
- 覆盖Bot并发、导入、蒸馏与回归测试用例
- **并发为测试重点**: 20+ Bot同时在线时的连接稳定性与响应延迟需可复现
- 导入QQ/微信记录后，需能生成至少1个可上线Bot

## 4. 聊天记录蒸馏 — 多机器人Skill方案

### 4.1 核心理念
不训练大模型，而是从真实聊天记录中提取**语言风格和情绪模式**，封装为可复用的Skill（系统提示词 + 少样本示例）。每个Skill赋予机器人独特的说话方式、情绪表达和用词习惯。通过不同API Key接入不同Skill，实现多个风格迥异的机器人同时在线聊天。

### 4.2 蒸馏流程
```
30天聊天记录 → 按用户聚类 → 提取语言特征 → 生成Skill配置 → 注册机器人 → 在线互动
```

**蒸馏输出要求**:
- 生成可直接上线的Skill配置（System Prompt + Few-shot）
- 可选生成“人设摘要”（口头禅/情绪倾向/回复节奏）用于前端展示
- 支持覆盖导入场景（文件导入）与在线场景（数据库蒸馏）

**语言特征提取**（不涉及模型训练，纯数据分析）:
- **情绪分布**: 统计用户常用情绪词、感叹词、语气助词，构建情绪向量
- **句式模式**: 分析句子长度分布、问句比例、惯用开头/结尾
- **用词偏好**: 提取高频词汇、惯用表达、Emoji使用习惯
- **对话节奏**: 回复速度、平均字数、是否偏好打断/承接

### 4.3 Skill 结构
```json
{
  "skill_id": "skill_001",
  "name": "温柔知心派",
  "emotion_profile": {
    "base_tone": "温和",
    "joy_ratio": 0.3, "care_ratio": 0.4, "humor_ratio": 0.1
  },
  "language_style": {
    "avg_sentence_len": 15,
    "use_emoji": true, "use_tone_words": true,
    "habit_openings": ["哈哈", "嗯嗯", "其实"],
    "habit_endings": ["呢", "哦", "呀"]
  },
  "system_prompt": "你是一个温柔体贴的聊天对象，说话轻声细语...",
  "few_shot_examples": [
    {"user": "今天好累", "assistant": "辛苦了呀，好好休息一下呢~"},
    {"user": "好无聊", "assistant": "哈哈，那我陪你聊聊天呀"}
  ]
}
```

### 4.4 多机器人共存架构
```
┌──────────────────────────────────────┐
│           Chatroom Server            │
│  ┌────────────────────────────────┐  │
│  │        Bot Manager             │  │
│  │  ┌──────┐ ┌──────┐ ┌──────┐   │  │
│  │  │Bot-1 │ │Bot-2 │ │Bot-N │   │  │  N ≥ 20
│  │  │SkillA│ │SkillB│ │SkillX│   │  │
│  │  └──┬───┘ └──┬───┘ └──┬───┘   │  │
│  └─────┼────────┼────────┼────────┘  │
│        │        │        │           │
└────────┼────────┼────────┼───────────┘
         │        │        │
    ┌────▼──┐ ┌──▼──┐ ┌───▼───┐
    │API-Key│ │API-2│ │API-Key│   (不同厂商/不同Key)
    │Skill A│ │Skill│ │Skill X│
    └───────┘ └─────┘ └───────┘
```

- 每个机器人 = 1个Bot账号 (is_bot=true) + 1个API Key + 1个Skill配置
- 机器人通过WebSocket模拟普通用户接入，消息路由与普通用户一致
- 不同API Key可来自同一厂商不同项目，或不同厂商，天然隔离互不影响
- Bot Manager负责: 注册/下线机器人、调度消息、失败重试、熔断降级

### 4.5 20机器人同时在线保障

| 策略 | 说明 |
|------|------|
| 连接池复用 | 每个机器人独立WebSocket长连接，心跳30s |
| API并发控制 | 信号量限制单Bot最大并发推理=1，避免消息风暴时API过载 |
| 熔断降级 | 连续3次API超时/错误 → 该Bot暂时静默30s，不影响其他Bot |
| 消息排队 | Bot收到多条消息时排队处理，超出队列容量(10条)丢弃最旧 |
| 错误隔离 | 每个Bot独立的API Key和配置，一个Bot的API故障不影响其余19个 |
| 监控告警 | Bot错误率 > 5%/min → 告警通知 → 自动下线该Bot |

### 4.6 导入QQ/微信聊天记录要求
- 优先支持JSON/TXT导出格式，并提供可解析的最小字段：`sender`、`content`、`timestamp`
- 需提供字段映射/角色映射（如群聊昵称 → 统一发送者ID）
- 导入后支持选择“生成Skill”或“仅存档/检索”

### 4.7 预期指标
- 同时在线Bot数: ≥ 20
- 单Bot回复延迟: 消息接收 → 推理 → 推送 < 2s (P95)
- Bot错误率: < 0.1% (API故障+超时+异常 / 总消息数)
- 20个Bot同时活跃时，系统CPU增加 < 30%

## 5. 技术约束
- 后端: Spring Boot 3.2 + Java 17
- 前端: Vue 3 + Element Plus
- 数据库: MySQL 8.0 (Demo使用H2)
- 通信: WebSocket (STOMP) + REST API
- 缓存: Redis (生产环境)
- 消息队列: RabbitMQ (生产环境)
